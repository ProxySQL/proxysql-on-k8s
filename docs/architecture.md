# Architecture

How the ProxySQL operator works under the hood — what it owns, how it
talks to ProxySQL, and the reasoning behind the major design choices.

## Components

```
┌────────────────────────────────────────────────────────────────────┐
│                  proxysql-operator (single Deployment)             │
│  ┌────────────────────────┐    ┌─────────────────────────────────┐ │
│  │ ProxySQLCluster        │    │ ProxySQLConfig                  │ │
│  │ controller             │    │ controller                      │ │
│  │                        │    │                                 │ │
│  │ owns: STS, Svc, Secret,│    │ owns: nothing                   │ │
│  │       CM, PDB, [SM]    │    │ side-effect: SQL to admin port  │ │
│  └────────────────────────┘    └─────────────────────────────────┘ │
│         │                              │                           │
│         │ uses                         │ uses                      │
│         ▼                              ▼                           │
│  ┌────────────────────────┐    ┌─────────────────────────────────┐ │
│  │ internal/controller/   │    │ internal/proxysqlclient         │ │
│  │   builders             │    │  Executor interface + sync.go   │ │
│  │ (pure desired-state)   │    │  (DELETE/INSERT/LOAD/SAVE per   │ │
│  │                        │    │   admin table)                  │ │
│  └────────────────────────┘    └─────────────────────────────────┘ │
└────────────────────────────────────────────────────────────────────┘
                                        │
                                        ▼
                              ┌───────────────────┐
                              │ ProxySQL admin    │
                              │ port 6032 (MySQL  │
                              │ wire protocol)    │
                              └───────────────────┘
```

### Packages

| Package | Responsibility |
| --- | --- |
| [`api/v1alpha1/`](../operator/api/v1alpha1) | CRD types (`ProxySQLCluster`, `ProxySQLConfig`) — deep-copy generated by `controller-gen`. |
| [`internal/controller/`](../operator/internal/controller) | The two reconcilers. Read CRs, decide on desired state, apply it. |
| [`internal/controller/builders/`](../operator/internal/controller/builders) | Pure builders that turn a defaulted `ProxySQLClusterSpec` into K8s objects (`*corev1.Secret`, `*appsv1.StatefulSet`, …). Stateless; trivially unit-testable. |
| [`internal/proxysqlclient/`](../operator/internal/proxysqlclient) | Thin MySQL-wire client for ProxySQL's admin port + the `Sync` function that writes every admin table. The reconciler calls this; tests use a recording `Executor` fake. |
| [`cmd/main.go`](../operator/cmd/main.go) | Manager entry point. Standard controller-runtime wiring + leader election. |

## CRDs

### `ProxySQLCluster` — the pods

What it represents: a set of ProxySQL pods exposing some combination of
MySQL / PostgreSQL / admin / metrics ports.

Owned objects (all with `OwnerReference{controller=true, blockOwnerDeletion=true}`):

| Object | Name | Purpose |
| --- | --- | --- |
| `Secret` | `<cluster>` *or* `spec.auth.secretName` | `admin-password`, `radmin-password`, `monitor-password`. Operator-minted when `spec.auth.secretName` is empty; user-managed otherwise. |
| `ConfigMap` | `<cluster>` | Bootstrap `proxysql.cnf` — only enough to start the admin port. Backends, users, query rules all come from `ProxySQLConfig`. |
| `Service` | `<cluster>` | ClusterIP. Exposes the enabled protocol ports + metrics. |
| `Service` | `<cluster>-headless` | `ClusterIP=None`, `publishNotReadyAddresses=true`. Used as the StatefulSet's `serviceName` so pods get stable DNS for ProxySQL Cluster sync. |
| `StatefulSet` | `<cluster>` | The pods. Annotated with `proxysql.com/cnf-checksum` so a content change triggers a rolling restart. PVC template optional (`spec.persistence.enabled`). |
| `PodDisruptionBudget` | `<cluster>` | `minAvailable = replicas - 1` by default. Omitted when `replicas ≤ 1`. |
| `ServiceMonitor` (unstructured) | `<cluster>` | Only when `spec.metrics.serviceMonitor.enabled=true`. Best-effort: a missing prometheus-operator CRD does **not** fail the reconcile; it just sets a `ServiceMonitorReady=False` condition. |

Status: `ObservedGeneration`, `Replicas`, `ReadyReplicas`, `AdminSecretName`,
and the standard `Available` / `Progressing` / `Degraded` conditions
following the K8s API conventions.

### `ProxySQLConfig` — the declarative configuration

What it represents: the set of `mysql_servers`, `mysql_users`,
`mysql_query_rules`, `mysql_replication_hostgroups`, `pgsql_servers`,
`pgsql_users`, `pgsql_query_rules`, `proxysql_servers`, and global
variables (`admin_variables` / `mysql_variables` / `pgsql_variables`)
that should be applied to the target `ProxySQLCluster`.

It owns **no** K8s objects. The side effect of reconciling it is
SQL writes to each ready replica's admin port. Status:
`ObservedGeneration`, `LastAppliedHash` (SHA-256 of the resolved
desired state), `LastSyncTime`, `SyncedReplicas`, and `Ready` /
`Progressing` / `Degraded` / `ClusterFound` conditions.

## Reconcile loops

### `ProxySQLClusterReconciler`

```
fetch CR
  └─► resolve admin Secret
        ├─ exists?  → read passwords (backfill any missing keys when operator-owned)
        └─ missing? → mint 32-char hex passwords, create Secret (errors out if SecretName was set externally)

build a Builder with defaulted spec + passwords

ensure ConfigMap     (CreateOrUpdate; SetControllerReference)
ensure Service       (preserves immutable ClusterIP/s on update)
ensure HeadlessSvc
ensure StatefulSet   (cnf-checksum annotation; selector immutable post-create)
ensure PDB           (or delete operator-owned one when replicas≤1 / disabled)
ensure ServiceMonitor (best-effort; condition-surfaced)

update status from the live StatefulSet
```

Triggers (`SetupWithManager`):

- `For(ProxySQLCluster)` — the CR itself
- `Owns(StatefulSet, Service, Secret, ConfigMap, PodDisruptionBudget)` — re-reconcile when an owned object drifts

### `ProxySQLConfigReconciler`

```
fetch CR
  └─► resolve clusterRef → ProxySQLCluster in same ns
        └─ missing? → ClusterFound=False, requeue 5s

read cluster admin Secret to obtain admin password
  └─ missing/incomplete? → condition + requeue

resolve user passwords (mysqlUsers/pgsqlUsers SecretKeyRef → Secret data)

discover ready pods via labels: proxysql.com/cluster=<cluster>, PodReady=True

hash(resolved Desired) → compare to status.LastAppliedHash
  ├─ match AND every ready replica was synced last time AND generation unchanged
  │   → short-circuit; requeue 30s safety net
  └─ otherwise → fan out

for each pod IP:port:
  open sql.DB to admin port (admin/<admin-password>)
  for each table:
    DELETE FROM <table>
    INSERT INTO <table> (...rows...)
    LOAD <X> TO RUNTIME ; SAVE <X> TO DISK

write status (LastAppliedHash, LastSyncTime, SyncedReplicas, conditions)
requeue 30s on full success / 5s on partial
```

Triggers (`SetupWithManager`):

- `For(ProxySQLConfig)` — the CR itself
- `Watches(ProxySQLCluster, …)` — re-reconcile every config pointing at a cluster when that cluster changes (replica count, admin secret rotation, etc.)
- `Watches(Pod, …)` — re-reconcile when a ProxySQL pod becomes Ready or restarts. A freshly-restarted pod has empty runtime tables; this watch ensures it gets config pushed promptly without waiting for the 30s safety requeue.

## Why this design

### Why a separate `ProxySQLConfig` CRD?

The cluster-shape and the runtime configuration have different change
cadences. The shape (replicas, image, resource requests) changes
rarely; the configuration (backends, users, query rules) changes
constantly as backends are added or rerouted. Splitting them:

- Keeps the cluster reconciler idempotent and cheap (no SQL writes on
  every change).
- Lets multiple configs target one cluster (multi-tenant scenarios) or
  one config target many clusters in the future.
- Gives an obvious place to surface `SyncedReplicas` and per-config
  conditions, separate from cluster health.

### Why write-to-all instead of letting ProxySQL Cluster sync handle it?

ProxySQL has a built-in cluster-sync feature. We *use* it (the operator
populates `proxysql_servers` and toggles `cluster_*` admin variables
when replicas > 1) but the **operator still writes to every replica
directly**. Two reasons:

1. **Accurate `SyncedReplicas` status.** A user reading `kubectl
   describe proxysqlconfig` wants to know how many pods actually have
   the new config — not whether the operator "asked one pod" and trusts
   cluster sync to propagate. Writing to each replica makes the
   distinction observable.
2. **Faster convergence on pod restarts.** A pod that restarted in the
   last 30s has empty runtime tables and waits for the next cluster
   sync tick to refill. The operator's pod-watch shortcut writes the
   config immediately. Cluster sync still operates as the
   belt-and-braces backup.

### Why is the bootstrap `proxysql.cnf` rendered into a ConfigMap rather than a Secret?

The cnf has to contain the admin/radmin/monitor passwords for ProxySQL
to boot listening on its admin port. Today those passwords land in a
ConfigMap — which in Kubernetes has the same RBAC + base64 security
boundary as a Secret. The distinction is mostly conventional and ops
ergonomics (kubectl-get is human-readable). Switching to a Secret-backed
cnf via an entrypoint substitution pattern is tracked as a future
enhancement; the code paths are already isolated in `builders/configmap.go`.

### Why `*bool` for `Enabled` fields?

Go's `bool` zero value is `false`, and `omitempty` strips zero values
on marshal. With a plain `bool` field tagged `omitempty`, sending
`enabled: false` from YAML round-trips as no value, and the CRD's
`+kubebuilder:default=true` then re-defaults it to `true` — so a user
*literally cannot* disable persistence/metrics/PDB. `*bool` makes the
field tri-state (`unset` / `true` / `false`) which is the standard
K8s API idiom for this exact problem.

### Why two reconcilers in one binary?

They share enough plumbing (scheme, client, leader election) that
co-locating them is the obvious choice. Splitting them across binaries
would have us deploying two manager Deployments to do one job.

## Pod-level details

Each ProxySQL pod runs a single container (no sidecars) configured for
[Pod Security Standards `restricted`](https://kubernetes.io/docs/concepts/security/pod-security-standards/):

- `runAsNonRoot=true`, uid/gid 999
- `allowPrivilegeEscalation=false`
- `readOnlyRootFilesystem=true`
- `capabilities.drop=[ALL]`
- `seccompProfile.type=RuntimeDefault`

`/etc/proxysql/proxysql.cnf` is mounted from the operator-managed
ConfigMap (read-only). `/var/lib/proxysql` is either a per-pod PVC
(default) or an emptyDir (when `spec.persistence.enabled=false`).

Probes:

- Liveness + readiness both TCP-probe the admin port (6032). They do
  not run SQL queries — a deeper probe (e.g. "does
  `runtime_mysql_servers` have an entry?") would couple readiness to
  `ProxySQLConfig` having reconciled at least once, which we don't
  want. Pods with empty runtime tables are still Ready; the operator's
  pod-watch flips them to fully configured immediately.

## State diagram (ProxySQLCluster conditions)

```
                  (CR created)
                       │
                       ▼
              Available=False
              Progressing=True   ── pods coming up
                       │
                       │ (ready replicas == desired)
                       ▼
              Available=True
              Progressing=False  ── steady state

  any error  ──►  Degraded=True with Reason describing the failure
                  (e.g. AuthSecretError, ServiceMonitorReady=False)
```

`Available` ⇔ "the underlying StatefulSet has its full ready-replica
count". `Progressing` is on whenever the cluster isn't at steady
state. `Degraded` is set on specific recoverable error paths but
removed once the next reconcile succeeds.
